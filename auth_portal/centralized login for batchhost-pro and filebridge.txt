Centralized Login Portal for BatchHost-Pro & FileBridge System 
— Implementation Walkthrough

Overview
This document describes how to build a centralized authentication portal that sits in front of two separate systems — BatchHost-Pro and a second system FileBridge. After a user log in once, they are presented with a selection screen showing both systems as cards. Clicking Login on a system card triggers that system's splash screen and redirects to its dashboard.

User Roles
Both systems share the same three roles. Role-based access is enforced per system after redirect.

| Role | Description | Scope |
|---|---|---|
| `super_admin` | Full access across all organizations and both systems | Global |
| `organization_admin` | Full access within their assigned organization | Per-org |
| `organization_viewer` | Read-only access within their assigned organization | Per-org |

Multi-Organization Support
- Each user belongs to one organization (set at creation time by super_admin).
- `super_admin` users see all organizations' data in both systems.
- `organization_admin` and `organization_viewer` see only their org's data.
- The centralized login must pass the `organization_id` along with the session/token so each system can enforce its own org-level filtering.

Phase 1 — Centralized Auth Service

Shared Session / Cookie (Simpler, same server)
As the both systems run on the same domain, use a shared secret key and a common `users.json`. The central login sets a cookie (`auth_session`) and both Flask apps read it.

- Both apps must use the same `secret_key` for Flask sessions.
- Same-domain cookies are automatically shared.
- Simpler but less scalable.

Phase 4 — System-Specific Splash Screens
When the user clicks Login → on a system card, the flow is:

1. Call `POST /api/auth/launch {system: "BatchHost-Pro”} ` → get `launch_url`
2. Show the target system's splash screen (in-page overlay) for ~1.5 seconds
3. Navigate to `launch_url` (which will trigger the system's own splash on its login redirect)

JavaScript Launch Handler

```javascript
async function launchSystem(systemKey) {
  const btn = document.querySelector(`#card-${systemKey} .btn`);
  btn.disabled = true;
  btn.textContent = 'Launching…';

  try {
    const r = await fetch('/api/auth/launch', {
      method: 'POST',
      headers: { 'Content-Type': 'application/json' },
      body: JSON.stringify({ system: systemKey })
    });
    const data = await r.json();

    if (data.launch_url) {
      // Show system-specific splash
      showSplash(systemKey);
      // After splash animation (1.5s), navigate
      setTimeout(() => {
        window.location.href = data.launch_url;
      }, 1500);
    } else {
      flash(data.message || 'Launch failed', 'error');
      btn.disabled = false;
      btn.textContent = 'Login →';
    }
  } catch (e) {
    flash('Connection error', 'error');
    btn.disabled = false;
    btn.textContent = 'Login →';
  }
}
```

Phase 5 — BatchHost-Pro SSO Endpoint
Add a new route to the existing server.py of BatchHost-Pro System to accept the launch token and create a local session.

```python
@app.route('/auth/sso')
def sso_login():
    """Accept a launch token from the central auth portal."""
    token = request.args.get('token')
    if not token:
        return redirect(url_for('login_page'))

    # Verify token with the central auth service
    try:
        resp = requests.post(
            f"{CENTRAL_AUTH_URL}/api/auth/verify-token",
            json={"token": token},
            timeout=5
        )
        data = resp.json()
    except Exception:
        flash_msg = "SSO verification failed"
        return redirect(url_for('login_page'))

    if not data.get('valid'):
        return redirect(url_for('login_page'))

    user = data['user']
    # Establish Flask session exactly as normal login does
    session['user_id'] = user['id']
    session['role'] = user['role']
    session['organization_id'] = user['organization_id']
    session['login_at'] = time.time()

    # Mark web presence, log admin action, etc. (same as existing login)
    mark_web_presence(user['id'])

    # Show splash on BatchHost-Pro's dashboard
    return redirect(url_for('dashboard'))
```

> NOTE - The `show_splash` flag can be passed as a query param (`?splash=1`) so the dashboard template knows to show the splash on first load after SSO redirect.

Phase 7 — Backend: Central Auth `server.py`

Key Endpoints
```python
# POST /api/auth/login
@app.route('/api/auth/login', methods=['POST'])
def auth_login():
    data = request.json
    email = data.get('email', '').strip()
    password = data.get('password', '')

    # Load from shared users.json
    users = load_json(USERS_FILE)
    user = next((u for u in users if u['email'] == email and u['status'] == 'active'), None)
    if not user or not verify_pw(user['password'], password):
        return jsonify({'success': False, 'message': 'Invalid credentials'}), 401

    session['user_id'] = user['id']
    session['role'] = user['role']
    session['organization_id'] = user['organization_id']

    # Determine which systems this user can access
    allowed_systems = get_allowed_systems(user)

    return jsonify({
        'success': True,
        'redirect': '/select',
        'user': {
            'name': user.get('username'),
            'role': user['role'],
            'organization': get_org_name(user['organization_id'])
        },
        'systems': allowed_systems
    })


# POST /api/auth/launch
@app.route('/api/auth/launch', methods=['POST'])
@login_required
def auth_launch():
    system_key = request.json.get('system')
    user = _current_user()

    # Generate a short-lived one-use token
    token = generate_launch_token(user, system_key)
    system_config = SYSTEMS[system_key]

    return jsonify({
        'launch_url': f"{system_config['base_url']}/auth/sso?token={token}"
    })


# POST /api/auth/verify-token  (called by target systems)
@app.route('/api/auth/verify-token', methods=['POST'])
def verify_token():
    token = request.json.get('token')
    payload = consume_launch_token(token)  # one-use, 30s TTL
    if not payload:
        return jsonify({'valid': False}), 401
    return jsonify({'valid': True, 'user': payload['user']})
```

Phase 8 — Role-Based System Access Control
Control which systems a user can see on the selection screen:

```python
def get_allowed_systems(user):
    """Return list of systems a user is allowed to access."""
    role = user.get('role')
    allowed = []

    for key, config in SYSTEMS.items():
        # super_admin sees everything
        if role == 'super_admin':
            allowed.append({**config, 'key': key})
        # org roles: check if user's org is registered for this system
        elif user['organization_id'] in config.get('allowed_orgs', []):
            allowed.append({**config, 'key': key})

    return allowed
```

> TIP - Store per-system org allowlists in `settings.json` or a separate `systems.json` file for easy management without code changes.

User Flow Summary

```mermaid
sequenceDiagram
    participant U as User
    participant P as Auth Portal
    participant S as Target System (BatchHost)

    U->>P: Login (email/pw)
    P->>P: Validate against users.json
    P->>U: Redirect to /select
    U->>P: Click "Login" on Target
    P->>S: Generate token
    P->>U: Redirect to S/auth/sso?token=...
    U->>S: Access SSO URL
    S->>P: Verify token (POST /verify-token)
    P-->>S: Return User Object
    S->>S: Create local session
    S->>U: Redirect to Dashboard
```

1. User opens: https://portal.yourdomain.com/login
2. Portal shows its own splash (once per session) - YourPortal
3. User enters email + password
4. Central auth validates credentials against shared users.json
5. Redirect to: /select
6. User sees two cards: BatchHost-Pro | FileBridge. Each card shows: Logo, Name, Description, Role badge, Org name, "Login ->" button
7. User clicks "Login ->" on BatchHost-Pro
8. Portal shows BatchHost-Pro splash screen (1.5s, blue theme)
9. Browser navigates to: https://batchhost.yourdomain.com/auth/sso?token=<launch_token>
10. BatchHost-Pro verifies token with portal
11. BatchHost-Pro creates local session, redirects to /dashboard
12. BatchHost-Pro dashboard loads (optionally with its own splash on first load)

Security Considerations
CAUTIONS: 
- Launch tokens must be one-use and expire in ≤ 30 seconds to prevent replay attacks.
- Communicate between auth portal and target systems over HTTPS only.
- Use the existing CSRF protection pattern from server.py of BatchHost-Pro System on all state-mutating endpoints.
- Never expose the shared `SECRET_KEY` in version control — use environment variables.
- Rate-limit `/api/auth/login` using the existing `rate_limit` decorator from rate_limiter.py of BatchHost-Pro System.

1. Where does System B live? — Same server as BatchHost-Pro

4. Do all users have access to both systems? — Yes
5. Session strategy — This system uses stateful, server-side Flask sessions
